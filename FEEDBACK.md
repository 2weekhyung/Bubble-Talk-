# 🚀 Bubble Talk 프로젝트 피드백 및 개선 로드맵

본 문서는 현재 진행 중인 'Bubble Talk' 프로젝트를 이직용 포트폴리오로서 완성도를 높이기 위한 기술적 피드백과 개선 방향을 정리한 리포트입니다.

## 1. 🌟 현재의 강점 (Good Points)
- **실무 지향적 기술 스택:** Redis(ZSet, Value), Querydsl, WebSocket 등 백엔드 개발자로서의 기술적 깊이를 보여줄 수 있는 조합입니다.
- **효율적인 데이터 구조 활용:** 랭킹 시스템에 Redis `ZSet`을 활용하고, `increment`를 이용해 Rate Limiting을 구현한 점은 매우 실무적인 접근입니다.
- **깔끔한 계층 구조:** DTO 분리와 Custom Repository 패턴 적용을 통해 코드의 확장성을 고려한 설계가 돋보입니다.

---

## 2. 🛠 주요 개선 제안 (Critical Feedback)

### 2.1 성능 최적화 (Performance)
- **[채팅] 금칙어 필터링 성능 개선:**
  - **문제:** 메시지 전송 시마다 DB에서 금칙어 목록을 조회함 (`findAll`). 메시지량이 많아질 경우 DB 병목 발생.
  - **해결책:** 금칙어 목록을 Redis에 캐싱하거나, 서버 기동 시 메모리에 로드(Set/Trie 구조)하여 사용.
- **[투표] DB 쓰기 부하 분산:**
  - **문제:** 투표마다 Redis 점수 증가와 DB `save`를 동시에 수행.
  - **해결책:** 투표 로그는 Redis에 우선 기록하고, 일정 시간마다 배치 처리하거나 정산 시점에 일괄 반영(Write-behind 전략) 고려.

### 2.2 안정성 및 동시성 (Stability)
- **[메뉴] 중복 생성 방지 (Race Condition):**
  - **문제:** `findByMenuName` 조회 후 `save` 하는 찰나에 다른 요청이 들어올 경우 동일 메뉴가 중복 생성될 가능성 있음.
  - **해결책:** DB 유니크 제약 조건 설정 및 `upsert` 로직 적용, 혹은 Redis 분산 락(Redisson)을 활용한 원자성 보장.
- **[정산] 스케줄러 안정성:**
  - **문제:** 정산(`syncRedisToDb`) 중 오류 발생 시 데이터 유실 또는 중복 정산 위험.
  - **해결책:** 트랜잭션 범위 재점검 및 정산 완료 플래그 도입.

### 2.3 코드 구조 개선 (Refactoring)
- **Redis 키 관리 중앙화:** 흩어져 있는 문자열 키(`lunch:ranking:`)를 `RedisKeyManager` 또는 Enum으로 관리하여 오타 방지 및 유지보수성 향상.
- **전역 예외 처리:** `@RestControllerAdvice`를 도입하여 비즈니스 예외(이미 투표함, 존재하지 않는 메뉴 등)를 표준화된 에러 응답으로 제공.

## 3. ✅ 포트폴리오 완성도 체크리스트
- [ ] **테스트 코드 작성:** 핵심 비즈니스 로직(MenuService, ChatService)에 대한 JUnit5 단위 테스트 추가.
- [ ] **Spring Security 고도화:** 현재의 금칙어 관리에서 더 나아가, JWT 또는 세션 기반의 실제 사용자 인증/인가 추가.
- [ ] **API 문서화:** Swagger(SpringDoc)를 적용하여 프론트엔드와 소통 가능한 인터페이스 정의.
- [ ] **인프라 구성:** `Dockerfile` 작성 및 GitHub Actions를 활용한 간단한 CI/CD 파이프라인 구축.
